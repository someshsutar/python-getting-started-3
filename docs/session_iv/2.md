# Unit Tests with `unittest` and `pytest` in Python

Unit testing is the process of testing **individual units (functions, methods, or classes)** of a program in isolation to ensure they work correctly. Python provides the built-in **`unittest`** framework, while **`pytest`** is a popular third-party testing framework known for its simplicity and powerful features.

---

# What is Unit Testing?

A **unit** is the smallest testable piece of software, such as:

* A function
* A method
* A class

The goal is to verify that each unit behaves as expected for different inputs.

### Benefits of Unit Testing

* Detects bugs early
* Prevents regressions
* Makes refactoring safer
* Improves code quality
* Serves as documentation for code behavior

def add(a, b):
    return a + b

    return a * b

        raise ValueError("Division by zero is not allowed")

# Unit Testing with `unittest`

## What is `unittest`?

* Built into Python
* Inspired by Java's JUnit
* Uses classes and assertion methods
* Suitable for large enterprise projects

No installation is required.

---

## Basic Structure

```python
import unittest

class TestClass(unittest.TestCase):

    def test_method(self):
        pass

if __name__ == "__main__":
    unittest.main()
```

---

# Example

**test_calculator_unittest.py**

```python
import unittest
from calculator import add, subtract, multiply, divide

class TestCalculator(unittest.TestCase):

    def test_add(self):
        self.assertEqual(add(5, 3), 8)

    def test_subtract(self):
        self.assertEqual(subtract(10, 4), 6)

    def test_multiply(self):
        self.assertEqual(multiply(4, 6), 24)

    def test_divide(self):
        self.assertEqual(divide(20, 5), 4)

    def test_divide_by_zero(self):
        with self.assertRaises(ValueError):
            divide(10, 0)

if __name__ == "__main__":
    unittest.main()
```

# Common Assertion Methods

| Assertion            | Description          |
| -------------------- | -------------------- |
| assertEqual(a, b)    | Checks equality      |
| assertNotEqual(a, b) | Checks inequality    |
| assertTrue(x)        | Checks if True       |
| assertFalse(x)       | Checks if False      |
| assertIsNone(x)      | Checks None          |
| assertIsNotNone(x)   | Checks not None      |
| assertIn(a, b)       | Checks membership    |
| assertNotIn(a, b)    | Checks absence       |
| assertGreater(a, b)  | Greater than         |
| assertLess(a, b)     | Less than            |
| assertRaises()       | Expects an exception |

Example:

```python
self.assertGreater(20, 10)
self.assertIn("cat", "catapult")
self.assertFalse(False)
```

# Test Setup and Cleanup

Sometimes multiple tests require the same setup.

```python
import unittest

class TestExample(unittest.TestCase):

    def setUp(self):
        print("Setup before every test")
        self.number = 100

    def tearDown(self):
        print("Cleanup after every test")

    def test_add(self):
        self.assertEqual(self.number + 10, 110)

    def test_subtract(self):
        self.assertEqual(self.number - 20, 80)

if __name__ == "__main__":
    unittest.main()
```

Execution flow:

```text
setUp()

↓

Test Method

↓

tearDown()
```

---

# Class-Level Setup

```python
class TestDatabase(unittest.TestCase):

    @classmethod
    def setUpClass(cls):
        print("Connect to database")

    @classmethod
    def tearDownClass(cls):
        print("Close database")
```

Runs only once before and after all tests in the class.

# Unit Testing with `pytest`

## What is `pytest`?

`pytest` is a third-party testing framework that is:

* Easier to write
* More readable
* More powerful
* Widely used in industry

Install it with:

```bash
pip install pytest
```

# Basic Test

**test_calculator_pytest.py**

```python
from calculator import add, subtract, multiply, divide

def test_add():
    assert add(5, 3) == 8

def test_subtract():
    assert subtract(8, 3) == 5

def test_multiply():
    assert multiply(5, 6) == 30

def test_divide():
    assert divide(20, 5) == 4
```

Run:

```bash

```

Output

```text

=========================
```

Notice there are:

* No classes
* No `TestCase`
* No `main()`
* Simple `assert` statements

# Fixtures

Fixtures provide reusable setup and cleanup logic.

```python
import pytest

@pytest.fixture
def sample_numbers():
    return (20, 5)

def test_divide(sample_numbers):
    a, b = sample_numbers
    assert a / b == 4
```

Benefits:

* Reusable setup
* Cleaner tests
* Supports dependency injection

---

# Parameterized Tests

Instead of writing multiple similar tests:

```python
import pytest

@pytest.mark.parametrize(
    "a,b,result",
    [
        (1, 2, 3),
        (5, 7, 12),
        (10, 20, 30),
        (-1, 5, 4),
    ]
)
def test_add(a, b, result):
    assert a + b == result
```

This executes the same test four times with different inputs.

# unittest vs pytest

| Feature             | unittest                 | pytest                |
| ------------------- | ------------------------ | --------------------- |
| Installation        | Built-in                 | `pip install pytest`  |
| Test Style          | Class-based              | Function-based        |
| Assertions          | Many assertion methods   | Plain `assert`        |
| Fixtures            | `setUp()` / `tearDown()` | `@pytest.fixture`     |
| Parameterized Tests | Manual                   | Built-in              |
| Readability         | More boilerplate         | Very concise          |
| Plugins             | Limited                  | Rich plugin ecosystem |
| Learning Curve      | Moderate                 | Easy                  |

# Real-World Example: Employee Salary Calculator

**salary.py**

```python
def calculate_salary(hours, rate):
    if hours < 0 or rate < 0:
        raise ValueError("Negative values are not allowed")
    return hours * rate
```

### unittest

```python
import unittest
from salary import calculate_salary

class TestSalary(unittest.TestCase):

    def test_salary(self):
        self.assertEqual(calculate_salary(40, 500), 20000)

    def test_invalid_hours(self):
        with self.assertRaises(ValueError):
            calculate_salary(-10, 500)

if __name__ == "__main__":
    unittest.main()
```

### pytest

```python
import pytest
from salary import calculate_salary

def test_salary():
    assert calculate_salary(40, 500) == 20000

def test_invalid_hours():
    with pytest.raises(ValueError):
        calculate_salary(-10, 500)
```

# Best Practices

* Write one test for one behavior.
* Use descriptive test names (e.g., `test_divide_by_zero`).
* Test both valid and invalid inputs.
* Keep tests independent of each other.
* Avoid relying on external systems (databases, APIs) in unit tests—use mocks where needed.
* Organize tests in a dedicated `tests/` directory.
* Run tests automatically in your CI/CD pipeline.

---

# When to Choose Which?

* **Use `unittest`** when:

  * You want a framework included with Python.
  * You are maintaining legacy or enterprise codebases that already use `unittest`.

* **Use `pytest`** when:

  * You are starting a new project.
  * You want concise, readable tests.
  * You need advanced features like fixtures, parameterization, plugins, and detailed reporting.
