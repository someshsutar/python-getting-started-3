# Iterators vs Generators in Python

Both **iterators** and **generators** are used to produce values one at a time (lazy evaluation). The main difference is **how they are created**.

* An **iterator** is an object that implements the iterator protocol (`__iter__()` and `__next__()`).
* A **generator** is a simpler way to create an iterator using the `yield` keyword.

> **Key point:** Every **generator is an iterator**, but **not every iterator is a generator**.

---

# Comparison Table

| Feature                | Iterator                                     | Generator                            |
| ---------------------- | -------------------------------------------- | ------------------------------------ |
| Definition             | An object implementing the iterator protocol | A function that uses `yield`         |
| Creation               | Usually with a class                         | Using a function                     |
| Required Methods       | `__iter__()` and `__next__()`                | Only `yield`                         |
| State Management       | Managed manually                             | Managed automatically by Python      |
| Code Complexity        | More code                                    | Less code                            |
| Memory Usage           | Lazy (one value at a time)                   | Lazy (one value at a time)           |
| `next()` Support       | Yes                                          | Yes                                  |
| `for` Loop Support     | Yes                                          | Yes                                  |
| Raises `StopIteration` | Explicitly                                   | Automatically when function finishes |
| Best For               | Custom iteration logic                       | Sequential data generation           |

# Iterator Example

Here is a custom iterator that returns numbers from 1 to 5.

```python
class Counter:

    def __init__(self, limit):
        self.limit = limit
        self.current = 1

    def __iter__(self):
        return self

    def __next__(self):
        if self.current <= self.limit:
            value = self.current
            self.current += 1
            return value
        raise StopIteration


counter = Counter(5)

for number in counter:
    print(number)
```

Output

```text
1
2
3
4
5
```

Notice that you have to:

* maintain the current state
* increment it
* raise `StopIteration`

---

# Generator Example

The same logic becomes much simpler.

```python
def counter(limit):
    for i in range(1, limit + 1):
        yield i


for number in counter(5):
    print(number)
```

Output

```text
1
2
3
4
5
```

Python automatically:

* remembers the current position
* resumes execution after each `yield`
* raises `StopIteration` when done

---

# How Python Handles State

### Iterator

You manage the state manually.

```python
self.current += 1
```

### Generator

Python remembers the execution state automatically.

```python
yield value
```

When `next()` is called again, execution resumes immediately after the previous `yield`.

# Code Size Comparison

### Iterator

```python
class Squares:
    def __init__(self, n):
        self.i = 0
        self.n = n

    def __iter__(self):
        return self

    def __next__(self):
        if self.i < self.n:
            result = self.i ** 2
            self.i += 1
            return result
        raise StopIteration
```

About 15–20 lines of code.

---

### Generator

```python
def squares(n):
    for i in range(n):
        yield i ** 2
```

Only 3–4 lines.

---

# Memory Usage

Both are memory-efficient because they generate values on demand.

List:

```python
numbers = [i for i in range(1000000)]
```

Creates one million integers immediately.

Generator:

```python
numbers = (i for i in range(1000000))
```

Produces each integer only when needed.

A custom iterator can also be implemented lazily.

---

# StopIteration

### Iterator

You must raise it yourself.

```python
raise StopIteration
```

### Generator

Python raises it automatically when the generator function returns or reaches the end.

# `next()` Example

Iterator:

```python
counter = Counter(3)

print(next(counter))
print(next(counter))
print(next(counter))
```

Generator:

```python
gen = counter(3)

print(next(gen))
print(next(gen))
print(next(gen))
```

Both produce the same output:

```text
1
2
3
```

# Real-World Use Cases

### Use an Iterator when:

* You need fine-grained control over iteration.
* You're building a custom collection.
* Iteration behavior is complex or depends on object state.
* You're implementing your own data structure.

Example:

* Tree traversal
* Graph traversal
* Database cursor
* Custom container class

---

### Use a Generator when:

* Producing a sequence of values.
* Reading large files.
* Processing API streams.
* Working with pipelines.
* Creating infinite sequences.

Example:

```python
def fibonacci():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b
```

---

# Behind the Scenes

Generator:

```python
def numbers():
    yield 1
    yield 2
```

Internally behaves much like an iterator class.

Conceptually similar to:

```python
class Numbers:

    def __init__(self):
        self.state = 0

    def __iter__(self):
        return self

    def __next__(self):

        if self.state == 0:
            self.state = 1
            return 1

        elif self.state == 1:
            self.state = 2
            return 2

        raise StopIteration
```

Python generates this kind of state machine for you.

# Which One Should You Use?

| Situation                        | Recommended |
| -------------------------------- | ----------- |
| Generate values sequentially     | Generator   |
| Read a large file                | Generator   |
| Infinite sequence                | Generator   |
| Custom iterable object           | Iterator    |
| Complex traversal logic          | Iterator    |
| Need full control over iteration | Iterator    |

In day-to-day Python programming, **generators are generally preferred** because they are concise and easier to maintain. Use a custom iterator when you need behavior that doesn't naturally fit a generator function.

---

# Interview Questions

### 1. Is every generator an iterator?

**Yes.** A generator automatically implements the iterator protocol.

---

### 2. Is every iterator a generator?

**No.** A custom class implementing `__iter__()` and `__next__()` is an iterator but not a generator.

---

### 3. Which is faster?

For most sequential iteration tasks, generators are typically preferred because they avoid the overhead of manually managing state and are much simpler to write. Performance differences are usually small compared to the readability benefits.

---

### 4. Which uses less memory?

Both iterators and generators are lazy and generally use far less memory than building an entire list. Their memory usage is similar because neither stores the complete sequence by default.

# Summary

| Aspect      | Iterator               | Generator             |
| ----------- | ---------------------- | --------------------- |
| Built Using | Class                  | Function              |
| Keyword     | `__next__()`           | `yield`               |
| State       | Manual                 | Automatic             |
| Code Size   | Larger                 | Smaller               |
| Memory      | Lazy                   | Lazy                  |
| Best Use    | Custom iteration logic | Lazy value generation |

### Simple Memory Trick

* **Iterator** = **You drive the car**. You manually control the current position (`__next__()`), when to stop, and how iteration works.
* **Generator** = **The car has cruise control**. You simply place `yield` where values should be produced, and Python handles the rest.
